## 首次烟测发现与修复

生产管理员 UI 的 Codeforces 平台范围同步首先完成 6/6 个账号，证明确认对话框、平台范围请求、运行记录刷新和完成通知正常。

随后首次全体同步把 36 个已验证账号放入一次 Edge 调用。约 105 秒后前端收到非 2xx 网关错误；数据库中已有多平台运行记录，说明问题不是授权或数据加载，而是管理员入口没有复用计划任务已有的稳定游标分页。

PR #48 将管理员全体、平台和成员同步统一改为：

- 每页最多 6 个平台账号；
- 使用 `nextCursor` 串行请求后续页；
- 聚合每页的 requested、succeeded、queued、failed；
- 拒绝重复或不前进的游标；
- 首次操作和续页使用独立的有界限流键，既限制用户重复发起，也允许已开始的操作完成；
- 后续页失败时说明此前已处理的账号数。

PR 的 `verify`、`database-security` 和 `gitleaks` 全部通过。完整验证包括 Vitest 259 项、Deno 262 项、PostgreSQL 17 pgTAP 290 项、105 项浏览器流程与可访问性测试、构建、Lint 和 Edge 类型检查。

这证明全体入口完成了全部游标页，并能在部分数据源失败时返回完整聚合结果。

## 队列可见性与失败重试

为避免污染真实成员，使用临时管理员和可级联清理的夹具完成以下检查：

1. 创建一个未来四小时执行的 Codeforces `queued` 平台任务；
2. 创建一条带 `not_found` 的临时 AtCoder 失败运行；
3. 刷新生产同步中心后，当前任务队列显示“1 个活动任务”，包含排队状态、平台、手动触发、`0/3` 尝试、执行时间和 `timeout` 最近错误；
4. 失败运行显示结构化错误和启用的重试按钮；
5. 从最终生产 UI 点击重试；
6. 页面显示临时成员的 AtCoder 数据重试后仍失败，未把失败误报为成功；
7. 数据库确认临时成员共有 3 个任务、3 条运行且全部失败，其中 1 个任务的触发方式为 `retry`；
8. 独立队列夹具仍保持 `queued`，证明 UI 读取的是持久队列而不是瞬时本地状态。
