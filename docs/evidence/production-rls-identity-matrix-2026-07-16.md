# 生产 RLS 三身份矩阵（2026-07-16）

## 范围与安全边界

本次在生产 Supabase 中创建三个随机临时 Auth 用户，分别设为：

- 普通启用成员；
- 停用成员；
- 启用管理员。

三个账号均使用随机、一次性邮箱和密码。本文及命令输出没有记录邮箱、UUID、密码、JWT、API Key、QQ 或真实成员资料。测试仅创建临时 Profile 数据，不绑定平台账号、不触发第三方同步，也不修改现有成员。

## 验收结果

使用三个账号各自通过真实密码登录取得的 JWT 调用生产 PostgREST/RPC，11 项检查全部通过：

| 身份     | 检查                                                          | 结果 |
| -------- | ------------------------------------------------------------- | ---- |
| 普通成员 | 可以读取本人 Profile                                          | 通过 |
| 普通成员 | 不能读取另一临时管理员的私有 Profile                          | 通过 |
| 普通成员 | 不能读取私有审计数据                                          | 通过 |
| 普通成员 | 调用 `admin_list_members` 返回 HTTP 403                       | 通过 |
| 普通成员 | 尝试把本人角色改为管理员不生效，数据库角色仍为 `member`       | 通过 |
| 停用成员 | 仍可读取本人 Profile，符合“保留账号数据”的停用语义            | 通过 |
| 停用成员 | 修改本人 Profile 返回零受影响行，数据库值保持不变             | 通过 |
| 停用成员 | 调用 `admin_list_members` 返回 HTTP 403                       | 通过 |
| 停用成员 | 匿名公开成员视图不再包含该成员                                | 通过 |
| 管理员   | `admin_list_members` 返回 HTTP 200 并包含生产私有成员目录投影 | 通过 |
| 管理员   | `admin_list_audit_logs` 返回 HTTP 200 和脱敏审计投影          | 通过 |

该结果验证了普通成员、停用成员和管理员的真实生产 JWT/RLS 边界，而不只依赖空库 pgTAP 或 service role 查询。

## 清理与删除边界

Supabase Auth Admin 的常规删除端点被生产数据库的 `auth_users_0_require_fenced_deletion` 触发器拒绝，这是预期的最终安全边界：任何 Auth 删除必须来自持有目标绑定恢复租约的数据库 RPC。

测试清理因此使用与正式注销一致的数据库边界：

1. 临时管理员先降级为普通成员；
2. 每个账号分别获取绑定到目标用户的恢复租约；
3. 调用 `delete_auth_user_with_recovery_lease` 原子删除 Auth 用户及 Profile；
4. 再次查询 Auth 管理端和 Profile 表。

最终结果：

- 目标绑定租约删除成功：3/3；
- 临时 Auth 用户残留：0；
- 临时 Profile 残留：0；
- 未读取或记录任何现有成员身份。

## 尚未证明的范围

本次矩阵不替代以下生产验收：

- 由现有真实管理员通过最终 UI/RPC完成管理员角色交接；
- 配置外部恢复下限 GitHub Token 后，通过 `delete-account` Edge Function 完成用户自助注销；
- Storage 所有权和其他受控约束返回 `409`；
- 最终删除期间的双连接行锁阻塞；
- 删除后旧 access JWT 的全部私有 RLS/RPC 语义；
- 响应传输丢失后的最终状态对账。
