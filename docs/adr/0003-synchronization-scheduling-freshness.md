# ADR 0003：分组调度、同步去重与批次新鲜度

- 状态：已接受
- 日期：2026-07-14

## 背景

六个平台的更新成本和数据变化频率不同。同步任务由 GitHub Actions 触发，但 cron 是 best-effort，可能延迟；第三方网站也可能限流、变更结构或暂时不可用。失败不能把榜单值清零，也不能因为短暂延迟立即显示“已过期”。

## 决策

平台分为两个计划组：

| 平台                            | 北京时间          | GitHub cron     |
| ------------------------------- | ----------------- | --------------- |
| Codeforces、牛客、洛谷、AtCoder | 每天 07:00、19:00 | `0 11,23 * * *` |
| XCPC ELO、QOJ                   | 每周二 08:00      | `0 0 * * 2`     |

GitHub Actions 使用 service role 调用 `sync-stats`。函数规范化同步范围，将批量请求拆为单平台任务，再按平台并发上限调用 `sync-member`。XCPC 批次在 fan-out 前预热一次数据库共享缓存，避免每个成员重复下载完整数据集。

平台账号通过验证后立即触发一次该平台同步；新注册成员只在受限时间窗口内触发一次本人 XCPC ELO 自动匹配。普通成员不能手动同步。

活动任务使用数据库唯一键去重。`sync_jobs`/`sync_runs` 持久记录每次尝试；临时失败会写回到期时间，服务角色通过原子 claim 领取，超过 15 分钟未完成的尝试会被关闭并恢复。完整决策见 ADR 0006。

## 新鲜度语义

- 日更平台：错过下一次 07:00/19:00 批次并超过 2 小时宽限后，才显示过期。
- 周更平台：错过下一次周二 08:00 批次并超过 24 小时宽限后，才显示过期。
- 成功同步更新最新值、成功时间和快照。
- 失败同步保留最后成功值，记录结构化错误；不写入 0，不把旧缓存记成新成功。
- 带上游观测时间的成功快照按成员、平台和源时间幂等。

## 重试与并发

- 普通单平台任务最多执行 3 次，失败后分别等待 2 分钟和 4 分钟。
- QOJ 浏览器流程不自动重试，避免重复消耗 Firecrawl 会话和触发风控。
- XCPC 使用数据库租约、条件请求、刷新冷却和批量预热。
- Codeforces/AtCoder 并发 2，XCPC ELO 并发 4，牛客/洛谷/QOJ 并发 1；平台组之间串行。
- GitHub workflow 设置全局并发组，不取消正在进行的同步。

## 后果

优点：单个平台故障不会清空数据；页面状态与实际计划批次一致；高成本数据源不会按成员重复抓取。

代价：首次尝试仍由触发请求立即执行；如果触发请求在创建某个成员的平台任务前整体中断，该未创建任务要等下一次计划批次。已创建任务则可由持久队列恢复。

## 未采用方案

- 所有平台同频更新：浪费 QOJ/Firecrawl 额度，并增加 XCPC 大文件下载。
- 以“距离上次成功 N 小时”直接判过期：不能容忍 cron 的正常排队延迟。
- 失败写 0：会制造错误排名且丢失最后已知值。
- 无上限自动重试：会放大限流、WAF 和凭据错误。

## 后续条件

当成员规模使首次 fan-out 接近 Edge Function 时限时，将“先完整入队、再异步消费”作为下一步演进，并保留现有适配器结果契约和审计表。
