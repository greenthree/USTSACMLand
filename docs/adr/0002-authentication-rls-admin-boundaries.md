# ADR 0002：Supabase Auth、RLS 和受控管理员 RPC

- 状态：已接受
- 日期：2026-07-14

## 背景

网站同时服务访客、普通成员、停用成员和管理员。成员需要编辑本人姓名、QQ、年级、专业和平台账号；管理员需要查看私有资料、维护成员和平台绑定、录入统计并触发同步。仅靠前端隐藏按钮无法阻止越权调用。

## 决策

使用 Supabase Auth 管理邮箱、密码、会话、密码重置和刷新令牌。业务数据库不保存密码。

权限采用三层防护：

1. 前端路由守卫根据会话、`profiles.role` 和启用状态控制页面入口，仅作为用户体验层。
2. Postgres RLS 与列级权限限制浏览器可见行和可写字段，是普通请求的最终安全边界。
3. 管理员写操作通过 `SECURITY DEFINER` RPC 执行，函数内部重新检查管理员身份、目标状态、乐观锁和审计原因。

注册时姓名写入 Auth metadata，数据库触发器创建已启用的 profile，并建立服务端维护的 XCPC ELO 自动匹配账号。普通成员不能直接修改角色、启用状态、统计、快照、同步任务、审计日志或他人的资料。

`service_role` 只用于 Edge Functions、数据库运维和受控 GitHub Actions。它不进入浏览器，也不作为普通管理员登录凭据。

## 身份行为

| 身份         | 权限                                            |
| ------------ | ----------------------------------------------- |
| 未登录访客   | 读取公开成员视图、公开统计和公开页面            |
| 正常成员     | 读取/编辑本人允许字段，维护本人非 XCPC 平台账号 |
| 停用成员     | 可保留 Auth 账号，但业务编辑和后台访问被拒绝    |
| 管理员       | 读取后台数据，通过受控 RPC 执行管理操作         |
| service role | 由服务端执行同步、受限缓存 RPC 和部署操作       |

## 安全规则

- 所有 `SECURITY DEFINER` 函数固定 `search_path`，并显式撤销默认 `PUBLIC` 执行权。
- 核心表不向管理员提供直接 `FOR ALL` 策略；管理员直接更新不得绕过 RPC 的范围、并发检查和审计。
- QQ 默认不进入公开视图；公开错误不包含原始抓取响应和私有标识。
- 平台账号唯一约束阻止重复绑定，错误提示不泄露另一个成员身份。
- 账号状态和角色不能由 Auth metadata 自助提升。

## 后果

优点：权限错误即使出现在前端，也会被数据库拒绝；敏感管理操作具有稳定的审计入口。

代价：每个新增管理操作都需要设计 RPC、GRANT/RLS 和身份矩阵测试，不能只增加一个前端按钮。

## 未采用方案

- 仅依赖前端路由守卫：请求可以绕过页面直接调用 API。
- 给管理员核心表 `FOR ALL`：会绕过乐观锁、目标限制和审计语义。
- 自建密码表：增加哈希、重置、会话和泄漏风险，且重复 Supabase Auth 能力。

## 验证状态

仓库已包含静态权限审查和 pgTAP 身份矩阵，覆盖普通成员、停用成员、管理员、列权限、XCPC 自动账号和统计表写保护。生产邮箱确认、回调和会话恢复仍需按 ROADMAP 完成真实端到端验收；当前生产配置允许注册后直接建立会话。
