# ADR 0008：所有平台同步失败最多自动重试一次

- 状态：已接受，待生产部署与烟测
- 日期：2026-07-20
- 替代范围：ADR 0003、0005、0006 中关于平台尝试次数和 QOJ 禁止重试的条款

## 背景

原策略允许普通单平台任务最多执行 3 次，而 QOJ 只执行 1 次。两套规则增加了运维和告警解释成本；普通平台的第三次尝试也会延长最终失败时间。产品现在要求所有平台采用一致边界：首次可恢复失败后自动重试一次，第二次仍失败即终止。

## 决策

- 标准单平台任务统一使用 `max_attempts = 2`，包含 Codeforces、牛客、AtCoder、XCPC ELO、洛谷和 QOJ。
- 多平台直接请求仍不整体重试；正常批次和管理员同步由 `sync-stats` 拆成单平台任务，使每个平台独立获得一次重试机会，避免部分成功后重复全部平台。
- 只有适配器明确标记为 `retryable` 的可恢复错误才重新排队，例如网络错误、超时、临时限流和临时上游不可用。
- `not_found`、账号格式错误、认证凭据失效、结构变化、权限拒绝、数据校验失败和未分类内部错误立即终止，不自动重试。
- 第一次失败后固定等待 2 分钟，再由 `sync_jobs` 持久队列和数据库调度器领取第二次 attempt；第二次失败没有下一次到期时间。
- GitHub Actions、curl 和 `sync-stats -> sync-member` 传输层不直接重发批次请求。所有业务级重试只由数据库队列控制，避免双层重试。
- `complete_sync_job_attempt` 在行锁内检查当前 attempt，并以 `least(max_attempts, 2)` 限制遗留任务；stale worker 不能覆盖第二次 attempt 或最终状态。
- 迁移时把仍为 `queued` / `running` 的单平台任务归一到两次上限；历史完成记录保留原尝试数据。
- QOJ 的第二次 attempt 创建新的临时 Firecrawl 会话，并继续在 `finally` 中清理会话。不会复用 Cookie 或持久 profile。
- WebChat、缓存探针、中转站兼容性检查和其他付费 AI 请求不属于平台数据同步，继续禁止自动重试。

## 安全与成本边界

- QOJ 并发继续为 1；第二次 attempt 只能在第一次形成结构化可恢复错误后创建。
- 凭据错误、Cloudflare 结构变化和认证失效不重试，避免重复发送密码或放大风控。
- 同一逻辑任务继续使用活动去重键、attempt fencing、幂等快照和原子结果提交。
- 告警只在第二次失败或不可重试错误形成最终失败时发送，避免第一次临时失败产生重复告警。

## 后果

所有平台拥有一致、可解释的失败恢复语义，最终失败最多等待一次两分钟退避。相较旧策略，普通平台减少一次任务级尝试，QOJ 增加一次受控 Firecrawl 成本；通过串行并发、错误分类、额度监控和最终告警限制风险。
