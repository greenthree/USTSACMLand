# USTSACMLand 开发路线图

本文档按“先验证数据源，再搭建业务，再上线”的顺序推进。里程碑编号不等同于发布日期；每个阶段只有在验收条件满足后才能进入下一阶段。

## 范围基线

首个可用版本必须包含：

- Codeforces、牛客、AtCoder、XCPC ELO Rating。
- Codeforces、牛客、洛谷、QOJ 刷题数。
- 邮箱注册后自动登录、密码登录/找回。
- QQ、年级、专业、姓名和各平台账号维护。
- 平台账号验证、成员数据管理、手动同步、同步日志与数据源状态。
- GitHub Pages 公开站点和安全的服务端数据同步。

不属于首版阻塞项：比赛提醒、群机器人播报、题目明细浏览、战队训练计划、移动端原生应用。

## M0：技术验证与项目骨架

- [x] 创建 React + TypeScript + Vite 应用，配置代码检查、格式化和单元测试。
- [x] 创建 Supabase 配置和首个数据库 migration。
- [x] 从生产 Schema 生成 Supabase TypeScript 数据库类型并接入前端 Client。
- [x] 配置 GitHub Pages 子路径、静态资源、SPA `404.html` 回退和部署工作流。
- [x] 完成首次线上 Pages 发布和刷新路由验证。
- [x] 建立统一适配器接口：输入外部账号，输出标准化统计、源时间和错误码。
- [x] 完成 Codeforces `user.info` / `user.status` 分页验证，确认唯一题目去重规则。
- [x] 完成牛客 UID -> 唯一通过题数、UID -> 当前/最高 Rating 验证，并增加 WAF 时的 Firecrawl 回退。
- [x] 完成 AtCoder 历史 JSON 验证，覆盖有 Rating、未参赛和不存在用户。
- [x] 完成 XCPC ELO `data.js` 解析，并按“姓名 + 苏州科技大学”唯一匹配后保存稳定 `xcpc_*` ID。

- [x] 完成洛谷认证记录接口验证：按 UID 分页读取 Accepted 记录，仅统计 `P`/`B` 开头 PID 并去重，覆盖零题数、凭据失效、限流、结构变化和分页截断。
- [x] 完成 QOJ 可行性 Spike：确认需要认证浏览器会话，统计去重后的 Accepted 题目。

- [ ] 配置 QOJ 登录失败告警和 Firecrawl 用量监控。
- [ ] 为每个平台保存脱敏固定样本并编写契约测试。
- [ ] 编写架构决策记录：部署、认证、定时任务、秘密管理和 QOJ 备用方案。

M0 验收条件：

- GitHub Pages 测试页可访问，Supabase 测试环境可连通。
- 六个平台适配器均能对至少 3 个样本账号给出可解释结果，或明确采用已验证的备用方案。
- 凭据不进入浏览器包、日志或 Git 历史。
- QOJ 若 Firecrawl 自动登录无法稳定运行，必须在进入 M3 前确定独立浏览器 Worker 或管理员导入备用方案，并在 UI 标注来源与新鲜度。

预计工作量：4-7 个开发日，QOJ/洛谷的反自动化情况可能增加时间。

- [ ] 在生产 Supabase 验证邮箱确认、重定向和会话恢复流程。
- [x] 创建 `profiles`、`platform_accounts`、`platform_stats`、`stat_snapshots`。
- [x] 创建 `sync_jobs`、`sync_runs`、`announcements`、`audit_logs`。
- [x] 建立平台枚举、审核状态、同步状态和标准错误码。
- [x] 添加 QQ、平台账号唯一性约束和必要索引。
- [x] 编写 RLS：访客只读公开成员；成员只读写本人允许字段；管理员走受控策略/函数（已静态审查）。
- [x] 建立注册后自动创建 profile 的触发器。
- [x] 建立仅限 service role / SQL 管理员的一次性首管理员初始化流程。
- [ ] 为 RLS、唯一约束和角色提升攻击编写数据库测试。
- [ ] 定义账号删除、停用、数据保留和匿名化策略。

M1 验收条件：

- 未登录、普通成员、已停用成员、管理员四种身份的权限测试全部通过。
- 普通成员无法查看私有 QQ、修改他人资料、修改角色或直接写统计值。
- 密码只由 Supabase Auth 处理，业务表和日志中不存在密码字段。

预计工作量：4-6 个开发日。

## M2：注册、资料与平台绑定

目标：让成员注册后直接进入资料页并完成平台绑定。

- [x] 实现注册后自动登录、登录和忘记密码页面。
- [x] 实现资料表单：姓名、QQ、年级、可自由输入并联想的专业、平台账号，并在保存前读取当前用户数据。
- [ ] 牛客和洛谷字段使用 UID 校验；CF、AtCoder、QOJ 使用各自用户名规则。
- [x] XCPC ELO 不要求用户填写 ID，使用资料姓名和“苏州科技大学”自动匹配并显示验证状态。
- [ ] 实现平台账号存在性校验和标准化，如 CF Handle 大小写回写。
- [ ] 实现重复绑定提示，但不向普通用户泄露已绑定成员的隐私信息。
- [x] 移除成员审核流程；新 profile 自动启用，已停用状态继续由安全边界保留。
- [x] 资料修改不再触发成员重新审核，平台账号变更仍会重新验证。
- [x] 禁止普通成员手动同步；管理员同步、平台验证触发和计划任务保留活动任务去重与结果反馈。
- [x] 完成当前资料表单的移动端、加载和错误状态浏览器 QA。
- [ ] 补齐键盘访问、屏幕阅读器和生产数据端到端测试。

M2 验收条件：

- 新用户注册后自动建立会话并进入资料页，不需要管理员批准。
- 不存在或格式错误的平台账号不会被标记为已验证。
- 同一个平台账号不能被两个成员同时绑定。

预计工作量：5-7 个开发日。

## M3：同步服务与数据质量

目标：稳定生成榜单所需数据，并允许管理员定位问题。

- [x] 在 Edge Functions 中实现统一同步入口和逐平台适配器。
- [x] Codeforces：查询 Rating，分页获取提交，按题目 ID 去重。
- [x] 牛客：读取唯一通过题数和 Rating 历史，检测页面结构变化，并在 WAF 时回退 Firecrawl。
- [x] AtCoder：读取 Rating 历史，计算当前和历史最高 Rating。
- [x] XCPC ELO：下载解析数据集，按姓名和学校唯一匹配；零匹配与同校同名均返回结构化错误，唯一命中回写稳定 ID。
- [ ] XCPC ELO：增加低频缓存和失效策略。
- [x] 洛谷：使用 Supabase Secrets 中的 Cookie/CSRF 请求认证记录接口，分页读取全部 Accepted 记录，仅统计 `P`/`B` 开头 PID 并去重。
- [x] QOJ：使用 Supabase Secrets 中的专用账号，每次创建 Firecrawl `/interact` 临时会话并自动登录查询目标主页。
- [x] QOJ：完成生产真实账号 smoke test，并验证统计值和成功状态落库。
- [ ] QOJ：继续演练密码错误、Cloudflare 和 Firecrawl 限流路径。
- [ ] 保存最新统计和历史快照；同一源数据时间重复执行需幂等。
- [x] 实现 HTTP 超时/有限重试、四成员批次和管理员/计划任务活动任务去重。
- [ ] 完成队列最大重试、指数退避和跨平台并发限额。
- [x] 区分 `not_found`、`auth_expired`、`rate_limited`、`schema_changed`、`timeout` 等结构化错误码。
- [x] 实现 GitHub Actions 分组定时与手动触发：日更组北京时间 07:00/19:00，周更组每周二 08:00。
- [x] 平台账号验证通过后立即同步该平台；同步失败不回滚验证状态。
- [ ] 增加同步失败通知。
- [x] 添加计划批次新鲜度规则：日更平台错过最近 07:00/19:00 批次、周更平台错过最近周二 08:00 批次并超过执行宽限后才标记过期，同时区分正常、已过期和不可用。
- [ ] 添加适配器契约测试、集成测试和模拟故障测试。

建议同步频率：

| 数据源                             | 默认频率                  | 数据过期阈值 | 说明                                       |
| ---------------------------------- | ------------------------- | ------------ | ------------------------------------------ |
| Codeforces / 牛客 / 洛谷 / AtCoder | 每天 07:00、19:00（北京） | 14 小时      | 同一平台组按成员单次调度，保留排队延迟余量 |
| XCPC ELO / QOJ                     | 每周二 08:00（北京）      | 192 小时     | 源数据更新较慢或查询成本较高               |

GitHub Actions cron 使用 UTC：日更组为 `0 11,23 * * *`，周更组为 `0 0 * * 2`。定时触发是 best-effort，实际启动可能略有延迟。

M3 验收条件：

- 定时同步连续运行 7 天，无秘密泄漏、无限重试或重复快照。
- 单个平台故障不会阻止其他平台更新。
- 抓取失败保留最后成功值并标记过期，不会将统计清零。
- 管理员能从日志定位成员、平台、错误类型和最近成功时间。

预计工作量：7-12 个开发日。

## 主要风险清单

| 风险                          | 影响                                 | 应对                                           |
| ----------------------------- | ------------------------------------ | ---------------------------------------------- |
| 把 GitHub Pages 当作完整后端  | 密码和 Cookie 泄漏，无法安全管理用户 | 静态前端与 Supabase/Edge Functions 分离        |
| 牛客、洛谷、QOJ 页面结构变化  | 统计中断或解析错误                   | 独立适配器、固定样本契约测试、结构错误告警     |
| QOJ 登录或 Firecrawl 作业失败 | 批量同步失败                         | 专用账号、完整登录健康检查、失败告警和受控轮换 |
| 第三方限流或封禁              | 数据延迟、IP 被限制                  | 分批、缓存、低频同步、退避和明确 User-Agent    |
| 同名 XCPC ELO 选手            | Rating 绑定到错误成员                | 姓名和学校必须唯一命中；同校同名时拒绝自动绑定 |
| 平台账号重复或冒领            | 榜单身份错误                         | 唯一约束、管理员审核，后续增加账号所有权验证   |
| 抓取失败被写成 0              | 排名错误且难以发现                   | 保留最后成功值，单独记录错误和新鲜度           |
| 公开真实姓名和 QQ             | 隐私风险                             | QQ 默认私有；上线前确认姓名展示范围和删除流程  |
| 免费托管额度或休眠            | 同步和登录不可用                     | 监控额度，保留迁移/备份方案，必要时升级服务    |

## 完成定义

任一功能只有同时满足以下条件才可标记完成：

- 行为符合验收标准，正常、空数据、失败和无权限状态均有处理。
- 关键逻辑有自动化测试，数据源解析有固定样本契约测试。
- 不引入明文秘密、越权查询或未审计的管理员写操作。
- 文档、数据库 migration 和生成类型与实现保持一致。
- 在 GitHub Pages 生产构建中验证，而不只在本地开发服务器运行。
